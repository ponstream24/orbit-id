# Type field guidance

[English](../en/type-registry.md)

Type は物理テーブルではなく、長期間意味が変わらない **論理エンティティ種別** を表す 6-bit フィールド
（`0..63`）です。状態、権限、ロール、テーブル分割などの実装詳細は Type に含めません。

## 所有権

**Orbit ID はエンティティ名のグローバルカタログを公開しません。**

どの数値が何を意味するかは、各 deployer / 組織が自前の Type マップとして持ちます。そのマップは
プロダクトの非公開ドキュメントや内部設定に置き、**このリポジトリにプロダクト固有の Type 名や割当表を
載せないでください。**

ワイヤ形式の相互運用に必要なのは Type 名の共有ではなく、値が `0..63` であることと、あるデプロイで
永続データに使った値の意味を後から付け替えないことです。

## 公開仕様上の規則

- 値の範囲は `0..63` です。
- 値 `0` は **予約** です。実装は `generate(0)` を拒否しなければなりません。発行 ID 以外の
  sentinel / 「未指定」用途に限って使ってよいです。
- `1..63` はアプリケーション側の割当に使えます。
- あるデプロイの永続 / 共有データで一度使った値の意味は、そのデプロイ内で変更・再利用してはなりません。
  付け替えるより、その値を使わない（引退）方が安全です。
- 実験用の割当を、別 Type マップの共有本番データと混ぜてはなりません。
- 新しい Type は、**自組織のマップ**で既存 Type では表現できない永続的な identity boundary がある
  場合だけ追加します。
- `USER` / `ADMIN` のような変更可能なロールや、`ACTIVE` / `DELETED` のような状態を Type に
  割り当てないでください。

## モデリング指針

エンティティが後から権限やロールを得ても、既存 ID の Type は変えません。別の論理エンティティが
必要なら別 Type 値で ID を新規発行し、データモデル上で関連付けます。

ライブラリは Type を整数として扱います。名前付き定数は仕様ではなくアプリケーションコードに置きます。
